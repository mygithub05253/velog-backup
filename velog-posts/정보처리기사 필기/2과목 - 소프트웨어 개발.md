# 소프트웨어 테스트
- 소프트웨어 테스트에서 검증과 확인을 구별하면 찾고자 하는 결함 유형 명확히 하는 데 도움이 됨
## 검증 (Verification)
- 소프트웨어 개발 과정을 테스트하는 것
- 제품이 명세서대로 완성되었는지 검증하는 단계
- 개발자의 시각에서 제품의 생산 과정을 테스트하는 것
- 작업 제품이 요구 명세의 기능, 비기능 요구사항을 얼마나 잘 준수하는 지 측정
## 확인 (Validation)
- 소프트웨어가 고객의 요구사항과 기대를 충족하는 지 확인하는 단계
- 사용자의 요구사항을 잘 수행하고 있는 지 사용자의 시각에서 생산된 제품의 결과를 테스트하는 것
# 소프트웨어 모델링 (Modeling)
- 개발팀이 응용 문제를 이해하는 데 도움을 줄 수 있다.
- 모델링은 소프트웨어 개발 전체 단계에서 사용된다.
- 개발될 시스템에 대하여 여러 분야의 엔지니어들이 공통된 개념을 공유하는 데 도움을 준다.
- 절차적인 프로그램을 위한 자료 흐름도는 프로세스 위주의 모델링 방법이다.
# 소프트웨어 결함 관리 프로세스
1. 결함 관리 계획
2. 결함 기록
3. 결함 검토
4. 결함 수정
5. 결함 재확인
6. 최종 분석 및 보고서 작성
# 소프트웨어 형상 관리 (SCM : Software Configuration Management)
- 개발 과정에서 ==SW 변경 사항을 관리==하기 위한 일련의 활동
- 개발 전체 단계에 적용
- 가시성과 추적성을 보장함으로써 소프트웨어의 생산성과 품질 향상 가능
- 중요성 : 변경 사항 추적/통제, 무절제한 변경 방지, 개발 진행 이력 확인
- 형상 관리 역할: 
	- 배포본 관리 용이
	- 불필요한 소스 수정 제한
	- 동일 프로젝트에 대해 여러 개발자 동시 개발 가능
- 형상 관리 도구 : CVS, SVN, Git
## 소프트웨어 형상 관리 종류

| 종류    | 설명                                                                                                                                                |
| ----- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| 형상 식별 | 형상 관리 계획을 근거로 형상 관리의 대상이 무엇인지 식별하는 과정<br>관리 대상에 이름/번호 부여 후 계층 구조로 구분<br>→ 수정/추적 용이                                                                |
| 형상 통제 | 식별된 형상 항목에 대한 변경 요구 검토 (기준선 반영될 수 있도록)<br>변경 요구는 즉시 처리되는 것이 아님<br>→ 변경 요구를 검토, 승인하여 적절히 통제<br>→ 현재의 베이스라인에 잘 반영될 수 있도록 조정<br>형상통제위원회 승인을 통한 통제 필요 |
| 형상 감사 | 기준선(Base line)의 무결성 평가 위해 확인/검증/검열 과정 진행<br>형상 관리 계획대로 형상 관리가 진행되고 있는 지, 형상 항목의 변경이 요구사항에 맞도록 제대로 이루어 졌는지 등을 살펴보는 활동                              |
| 형상 기록 | 형상 식별/통제/감사 작업 결과를 기록/관리하고 보고서 작성                                                                                                                 |
# 소프트웨어 버전 등록 용어
- 버전 등록 과정 : 가져오기 → 인출 → 커밋 → 동기화 → 차이

|         용어          | 설명                                                                  |
| :-----------------: | ------------------------------------------------------------------- |
| 저장소<br>(Repository) | 최신 버전 및 변경 내역 관련 정보 저장소                                             |
|  가져오기<br>(Import)   | 버전 관리 미진행 상태의 초기 저장소에 처음으로 파일을 복사                                   |
| 체크아웃<br>(Check-out) | 프로그램 수정을 위해 저장소에서 파일을 받아옴                                           |
|  체크인<br>(Check-In)  | 체크아웃한 파일을 수정한 후 저장소에 새로운 버전으로 파일을 갱신                                |
|   커밋<br>(Commit)    | 체크인 시 이전 갱신된 내용이 있는 경우 “충돌(Conflict)”을 알린 후 Diff 도구를 이용해 수정 후 갱신 완료 |
|   동기화<br>(Update)   | 저장소에 있는 최신 버전 상태로 자신의 작업 공간을 동기화                                    |
# 소프트웨어 버전 관리 도구

|         도구          | 설명                                                                                                                                                                                                                         |
| :-----------------: | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|        공유폴더         | 버전 관리 자료가 로컬 컴퓨터 내 공유 폴더에 저장/관리<br>개발자는 개발이 완료된 파일을 약속된 공유 폴더에 매일 복사<br>담당자는 해당 공유 파일을 본인 PC에 복사해 이상 유무 확인<br>EX) SCCS, RCS(Revision Control System), PVCS, QVCS                                                           |
|      클라이언트/서버       | 버전 관리 자료가 중앙 시스템(서버)에 저장/관리<br>자신의 PC(클라이언트)로 복사/작업 후 중앙 서버에 업로드<br>하나의 파일을 여러 개발자가 작업할 경우 경고 메시지 출력<br>서버 문제 발생 시 개발자 간 협업 및 버전 관리 작업 중단<br>EX) CVS, SVN(Subversion), Clear Case, CVSNT, CMVC, Perforce                   |
|       분산 저장소        | 원격 저장소와 개발자 PC의 로컬 저장소에 함께 저장/관리<br>원격 저장소에 문제 발생해도 로컬 저장소 이용해 작업 가능<br>로컬 저장소에 우선 반영(commit) 후 원격 저장소에 반영 (Push)<br>로컬 저장소에 작업하여 처리 속도 빠름<br>EX) Git, Bitkeeper, GNU arch, DCVS, Bazzar, Mercurial, TeamWare, Plastic SCM |
| SVN<br>(Subversion) | CVS 개선 버전 / 아파치 재단이 2000년에 발표<br>커밋(Commit)할 때마다 리비전(Revision) 1씩 증가<br>서버는 주로 유닉스 (UNIX) / 오픈소스로 무료 사용 가능<br>CVS의 한계인 파일이나 디렉터리의 이름 변경, 이동 가능                                                                             |
|       Git (깃)       | 원격 저장소 : 여러 사람들이 협업을 위해 버전을 공동 관리 장소<br>로컬 저장소 : 개발자들이 본인의 실제 개발을 진행하는 장소<br>브랜치(Branch) 이용 시 기존 버전 관리 틀에 영향 없이 다양한 기능 테스트 가능<br>파일의 변화를 스냅샷(Snapshot)으로 저장<br>버전의 흐름 파악 가능                                                |

# 페르소나 (Persona)
- 어떤 제품 혹은 서비스를 사용할 만한 목표 집단 내의 다양한 사용자의 유형을 대표하는 가상의 인물
- 마케팅, 디자인, 판매 등 다양한 개발 조건 간의 사용자 위한 상호 소통의 도구로 이용
- 사용자 중심의 UI를 만들기 위해 사용자를 분류하고 정의
## 페르소나 특징
1. 가상의 사용자 유형 : 실제 사용자를 기반으로 하지만, 특정 사용자 그룹을 대표하는 가상의 인물
2. 상세한 정보 포함 : 인구 통계, 성격, 목표, 습관, 고통점 등 사용자에 대한 상세 정보 포함
3. 사용자 중심 설계 : 사용자 중심의 제품 및 서비스 설계를 위한 도구
4. 공감 형성 : 개발팀 내 사용자에 대한 공감 형성 및 이해 증진
## 페르소나 활용 분야
1. 마케팅 : 타겟 고객층 설정 및 마케팅 전략 수립
2. 디자인 : 사용자 요구에 맞는 제품 및 서비스 디자인
3. 판매 : 고객과의 효과적인 커뮤니케이션 및 영업 전략 수립
4. 개발 : 사용자 중심의 기능 개발 및 UI/UX 디자인

# EAI (Enterprise Application Integration)
- 기업 내 각종 앱/플랫폼 간 정보 전달/연계/통합 등 상호작용 연동하는 모듈 연계 방식

|       방식       | 설명                                                                              |
| :------------: | ------------------------------------------------------------------------------- |
| Point to Point | 데이터 간 포인트-포인트 개별 연결<br>변경 및 재사용 어려움                                             |
|  Hub & Spoke   | 중앙(Hub) 시스템을 통한 데이터 전송 → 중앙 집중형 방식<br>확장/유지보수 유리<br>중앙 허브 장애 발생 시 전체 시스템에 영향을 줌 |
|  Message Bus   | 앱 사이에 미들웨어를 두고 처리<br>확장성 및 대용량 처리 가능                                            |
|     Hybrid     | Hub & Spoke와 Message Bus의 혼합형<br>데이터 병목 현상 최소화 가능                               |
# 애플리케이션 패키징 (소프트웨어 패키징)
- 개발이 완료된 소프트웨어를 고객에게 인도하기 위해 패키징하고, 설치 메뉴얼, 사용 메뉴얼 등을 작성하는 일련의 배포용 설치 파일을 만드는 작업
- 향후 관리 편의성을 위해 모듈화하여 패키징
- 사용자를 중심으로 진행하며, 사용자의 다양한 환경에서 설치할 수 있도록 패키징
- 사용자의 불편함을 줄이고 사용자의 편의성을 먼저 고려
- 신규 및 변경 개발 소스를 식별하고, 이를 모듈화하여 상용 제품으로 패키징
- 사용자의 편의성을 위해 메뉴얼 및 버전 관리를 지속적으로 수행
- 범용 환경에서 사용이 가능하도록 일반적인 배포 형태로 패키징 진행
- 주의 사항
	- 전체 내용을 포함
	- 고객 중심 (사용자 중심)
	- 모듈화
	- 버전 관리 및 릴리즈 노트 관리
# 애플리케이션 테스트
## 애플리케이션 테스트 기본 원리
- 테스트는 기본적으로 결함이 존재함을 밝히는 것 (무결함 증명 불가능)
- 완벽한 테스트는 근본적으로 불가능 (무한 경로, 무한 입력 불가능)
- 결함 집중
	- ==파레토(Pareto) 법칙 - 20%의 모듈에서 전체 결함 80% 발생==
	- 애플리케이션의 결함의 대부분은 소수의 특정한 모듈에 집중되어 존재
	- 결함은 발생한 모듈에서 계속 추가로 발생할 가능성이 높음
- ==살충제 패러독스== : 동일한 테스트 케이스에 대한 반복 테스트는 새로운 버그를 발견하지 못함
- 오류-부재의 궤변
	- 결함이 없다 해도 요구사항 미충족 시 품질 저하
	- 테스트를 통해 결함이 발견되지 않았다고 해서 소프트웨어에 결함이 없다고 확신할 수 없다.
	- 테스트는 결함을 발견하는 활동일 뿐, 결함이 없음을 보장하는 활동이 아니다.
- Brooks의 법칙
	- 지연되는 프로젝트에 인력 추가 투입 시 더 지연됨
	- 소프트웨어 프로젝트 일정이 지연된다고 해서 프로젝트 말기에 새로운 인원을 추가 투입하면 프로젝트는 더욱 지연되게 된다는 내용
## 애플리케이션 테스트 분류
### 확인 시험(Validation Test)
1. 알파 테스트
2. 베타 테스트
3. 블랙박스 테스트
### 프로그램 실행 여부

| 분류     | 설명                                                    |
| ------ | ----------------------------------------------------- |
| 정적 테스트 | 프로그램 실행하지 않고 명세서, 소스 코드만 분석<br>EX) 워크 스루, 인스펙션, 코드 검사 |
| 동적 테스트 | 프로그램 실행 후 오류 검사<br>EX) 화이트/블랙박스 테스트                   |
#### 화이트박스 테스트 (White Box Test)
- 모듈 안의 내용(작동) 직접 확인 가능 => 모듈의 논리적인 구조 체계적 점검 가능
- 내부의 ==논리적인 모든 경로를 테스트== => 프로그램의 내부 구조를 고려
- 소스 코드의 모든 문장을 한 번 이상 수행
- 논리적 경로 점검 (선택, 반복 수행) => 논리 흐름도 이용 가능
- 테스트 데이터를 선택하기 위해 ==검증 기준(Coverage)== 정함
##### 화이트박스 테스트 종류
1. 기초 경로 검사 (Base Path Test)
	- 대표적인 화이트박스 테스트 기법 (동적 테스트)
	- 싸이클 최대 한 번으로 측정 결과는 실행 경로의 기초를 정의하는 지침으로 사용
2. 제어 구조 검사
	- ==조건 검사 / 루프 검사 / 자료 흐름 검사== (Data Flow Test)
#### 블랙박스 테스트 (Black Box Test)
- 모듈 내부의 내용을 알 수 없음
- 프로그램의 구조를 고려하지 않음
- 소프트웨어 인터페이스에서 실시되는 테스트
- 사용자의 요구사항 명세를 보면서 구현된 기능에 중점을 두는 테스트
- SW 각 기능이 완전히 작동되는 것을 입증하는 테스트로 “기능 테스트”라고 함
- ==동치 분할 검사 / 경계값 분석 / 원인-효과 그래프 검사 / 비교 검사 / 오류 예측 검사==
##### 블랙박스 테스트를 이용해 발견할 수 있는 오류
1. 비정상적인 자료를 입력해도 오류 처리를 수행하지 않는 경우
2. 정상적인 자료를 입력해도 요구된 기능이 제대로 수행되지 않는 경우
3. 경계값을 입력할 경우 요구된 출력 결과가 나오지 않는 경우
##### 블랙박스 테스트 종류
1. 동치 분할 검사 (Equivalence Partitioning Testing)
	- 입력 자료에 초점을 맞춰 테스트 케이스를 만들고 검사하는 방법
	- 프로그램의 입력 조건에 중점을 두고, 어느 하나의 입력 조건에 대해 타당한 값과 그렇지 못한 값을 설정하여 해당 입력 자료에 맞는 결과가 출력되는 지 확인하는 테스트 기법
2. 경계값 분석 (Boundary Value Analysis)
	- 입력 조건의 중간값보다 경계값에서 오류가 발생할 확률이 높다는 점을 이용
	- 입력 조건의 경계값을 테스트 케이스로 선정하여 검사하는 방법
3. 원인-효과 그래프 검사 (Cause-Effect Graphing Testing)
	- 입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석
	- 효용성이 높은 테스트 케이스를 선정하여 검사하는 방법
4. 오류 예측 검사 (Error Guessing)
	- 과거의 경험이나 확인자의 감각으로 검사하는 방법
5. 비교 검사 (Comparison Testing)
	- 여러 버전의 프로그램에 동일한 테스트 자료를 제공하여 동일한 결과가 출력되는 지 검사하는 방법
### 테스트 기반

|  분류   | 설명                                                                   |
| :---: | -------------------------------------------------------------------- |
| 명세 기반 | 사용자 요구사항에 대한 명세를 빠짐없이 테스트 케이스로 구현하는지 확인<br>EX) 동등 분할 / 경계값 분석 (블랙박스) |
| 구조 기반 | SW 내부 논리 흐름에 따라 테스트 케이스 작성/확인<br>EX) 구문 기반 / 결정 기반 / 조건 기반 (화이트박스)   |
| 경험 기반 | 테스터의 경험을 기반으로 수행<br>EX) 에러 추정, 체크리스트, 탐색적 테스팅                        |
### 시각(관점) 기반

|          분류          | 설명                                                                                    |
| :------------------: | ------------------------------------------------------------------------------------- |
| 검증<br>(Verification) | ==개발자==의 시각에서 제품의 생산 과정 테스트<br>제품이 명세서대로 완성되었는지 검증하는 단계<br>EX) 단위/통합/시스템 테스트          |
|  확인<br>(Validation)  | ==사용자==의 시각에서 생산된 제품의 결과 테스트<br>사용자의 요구사항을 잘 수행하고 있는지 확인하는 단계<br>EX) 인수 테스트 (알파 / 베타) |
### 목적 기반

|         분류          | 설명                                          |
| :-----------------: | ------------------------------------------- |
|  회복<br>(Recovery)   | 시스템에 인위적 결함 부여 후 정상으로 회복되는 과정 확인            |
|  안전<br>(Security)   | 외부 불법 침입으로부터 시스템을 보호할 수 있는 지 확인             |
|   강도<br>(Stress)    | 과부하 시 SW 정상 구동 여부 확인                        |
| 성능<br>(Performance) | 실시간 성능 및 전체적인 효율성 진단<br>EX) 응답 시간, 업무 처리량 등 |
|  구조<br>(Structure)  | SW 내부 논리적인 경로 및 소스 코드 복잡도 평가                |
| 회귀<br>(Regression)  | SW 내 변경 또는 수정된 코드에 새로운 결함이 없음을 확인           |
|  병행<br>(Parallel)   | 변경 및 기존 SW에 동일한 데이터 입력 후 결과 비교              |
## 개발 단계에 따른 애플리케이션 테스트
### 단위 테스트 (Unit Test)
- 최소 단위 (모듈/컴포넌트) 기반 테스트
	- 모듈 : 소프트웨어를 이루는 기본 단위 (독립적 기능)
- 주로 구조 기반 테스트를 진행하며 기능성 테스트를 최우선으로 함
- 모듈의 동작 검증
#### 단위 테스트로 식별하거나 검출할 수 있는 오류 유형
1. 모듈 내부 로직 상의 오류나 계산 오류
2. 조건문 또는 반복문에서의 경계값 처리 오류
3. 입력 값에 대한 데이터 유효성 검증 실패
- 오답 보기 : 비트 손상이나 하드웨어 기반의 오류
### 성능 테스트
- 애플리케이션의 처리량, 응답 시간, 경과 시간, 자원 사용률에 대해 가상의 사용자를 생성하고 테스트를 수행함으로써 성능 목표를 달성했는 지 확인하고 테스트
### 통합 테스트 (Integration Test)
- 단위 테스트 후 모듈을 통합하는 과정에서 발생하는 오류 및 결함을 찾는 테스트 기법
- 모듈 통합 과정에서 각 모듈 간의 인터페이스 결함 검증
- 시스템을 구성하는 모듈의 인터페이스와 결합을 테스트하는 것
- 모듈 간의 인터페이스와 시스템의 동작이 정상적으로 잘되고 있는지를 빨리 파악하고자 할 때 상향식 보다는 하향식 통합 테스트를 사용하는 것을 권장

|         종류         | 설명                                                                                                                          |
| :----------------: | --------------------------------------------------------------------------------------------------------------------------- |
| 하향식<br>(Top-down)  | 상위 모듈에서 하위 모듈 방향으로 통합<br>종류 : 깊이 우선 통합법, 넓이 우선 통합법<br>초기부터 사용자에게 시스템 구조를 보여줌<br>==스텁(Stub)== : 모듈의 기능을 단순히 수행하는 도구 (시험용 모듈) |
| 상향식<br>(Bottom-up) | 하위 모듈에서 상위 모듈 방향으로 통합<br>하나의 주요 제어 모듈과 관련된 종속 모듈의 그룹인 ==클러스터(Cluster), 드라이버(Driver) 사용==<br>==스텁(Stub) 미사용==                |
#### 점진적 통합 방식 (빅뱅 테스트)
- 모든 모듈이 결합된 프로그램 전체가 테스트 대상
#### 비점진적 통합 방식
- 상향식 : 하위 → 상위로 통합해 가면서 테스트
- 하향식 : 상위 → 하위로 통합해 가면서 테스트
#### 혼합식 테스트 (Sandwich Test)
- 상향식과 하향식의 장점을 이용하는 방식 (상향식 + 하향식)
- 스텁(Stub)과 드라이버(Driver)의 필요성이 매우 높은 방식
#### 회귀 시험 (Regression Test)
- 수정한 부분이 소프트웨어의 다른 부분에 영향을 미치는지 테스트하여 소프트웨어 수정이 새로운 오류를 발생시키지 않았는지 확인
- 유형 : Retest All, Selective, Priority 기법
#### 상향식 통합 테스트 절차
1. 하위 모듈을 클러스터로 결합
	- 가장 낮은 수준의 모듈들을 기능적으로 묶어 클러스터를 구성
2. 상위 모듈에서 데이터 입출력을 확인하기 위해 더미 모듈인 드라이버 작성
	- 상위 모듈과의 인터페이스를 구현하는 드라이버를 만들어 상위 모듈의 기능을 시뮬레이션
3. 통합된 클러스터 단위로 테스트를 수행
	- 드라이버를 통해 클러스터에 데이터를 입력하고 출력 결과를 검증하여 클러스터의 기능을 테스트
4. 테스트가 완료되면 클러스터는 프로그램 구조의 상위로 이동하여 결합하고 드라이버는 실제 모듈로 대체
	- 테스트를 통과한 클러스터는 상위 모듈과 결합되고 드라이버는 실제 모듈로 교체
#### 하향식 통합 테스트
- 상위 모듈부터 테스트하고, 하위 모듈이 없을 경우 임시 모듈인 스텁(Stub) 사용
	- 상위 컴포넌트를 테스트하고 점증적으로 하위 컴포넌트 테스트
- 해당 방식에서는 상위 모듈이 주도하기 때문에 입력과 출력을 흉내내는 드라이버 필요 없음
- 종류 : 깊이 우선 방식, 너비 우선 방식
### 시스템 테스트 (System Test)
- 개발된 SW의 컴퓨터 시스템 내 작동 여부 점검
- 실제 사용 환경과 유사한 테스트 환경 → 기능적(블랙박스) 및 비기능적(화이트박스) 테스트 구분
- 전체 시스템의 기능 검증
### 인수 테스트 (Acceptance Test)
- 사용자의 요구사항 충족 여부 확인
- 소프트웨어 제품에 대한 요구사항이 제대로 이행되었는 지 점검하는 것이 주 목적
#### 인수 테스트 특징
- 일반적인 테스트 레벨의 가장 마지막 상위 레벨
- 테스트 환경을 실사용자 환경에서 진행하며 수행하는 주체가 사용자
#### 인수 테스트 종류

| 종류     | 설명                                    |
| ------ | ------------------------------------- |
| 알파 테스트 | 통제된 환경에서 사용자가 개발자와 함께 확인              |
| 베타 테스트 | 통제되지 않은 환경에서 ==개발자 없이== 여러 명의 사용자가 검증 |
# 테스트 케이스 (Test Case)
- 개발된 SW가 사용자의 요구사항을 잘 반영했는지 확인하기 위해 설계된 ==입력값(데이터), 실행 조건, 기대 결과== 등으로 구성된 테스트 항목에 대한 명세서
- 테스트의 목표/방향 결정 후 테스트 케이스를 작성함
- 테스트 케이스에는 일반적으로 시험 조건, 테스트 데이터, 예상 결과 포함
- 프로그램에 결함이 있더라도 입력에 대해 정상적인 결과를 낼 수 있기 때문에 결함을 검사할 수 있는 테스트 케이스를 찾는 것이 중요
## 테스트 케이스 자동 생성 기법
1. 입력 도메인 분석 (Input Domain Analysis)
	- 입력값의 범위를 나누어 자동으로 테스트 케이스 생성
2. 랜덤 테스트 (Random Test)
	- 무작위 입력값을 발생시켜 테스트 데이터 자동 생성
3. 자료 흐름도 (Data Flow Testing)
	- 변수 정의와 사용 관계를 분석해 테스트케이스 도출

- 특성 : 제한된 검증 / 수학적 기법 / 자동화 기능

# 외계인 코드 (Alien Code)
- 아주 오래되거나 참고 문서가 없어 유지보수 작업이 어려운 코드
# 나쁜 코드 (Bad Code)
- 다른 개발자가 로직(Logic)을 이해하기 어렵게 작성된 코드
## 나쁜 코드 종류
1. 스파게티 코드 : 처리 로직의 제어가 정제되지 않고 복잡하게 얽혀 있는 코드
2. 변수나 메소드에 대한 이름 정의를 알 수 없는 코드
3. 동일 로직이 중복되게 작성된 코드
# 클린 코드 (Clean Code)
- 깔끔하게 잘 정리된 코드
- 중복 코드 제거로 애플리케이션의 설계 개선
- 가독성 향상
- 버그를 찾기 쉬워지며, 프로그래밍 속도가 빨라짐
## 클린 코드 작성 원칙
1. 가독성
	- 누구나 코드를 쉽게 읽을 수 있도록 작성
	- 이해하기 쉬운 용어를 사용하고 들여쓰기 등을 활용
2. 단순성
	- 한 번에 한 가지 기능만 처리
	- 클래스/메소드/함수를 최소 단위로 분리
3. 의존성 배제
	- 다른 모듈에 미치는 영향을 최소화
	- 코드 변경 시 다른 부분에 영향이 없도록 작성
4. 중복성 최소화
	- 중복된 코드는 삭제하여 공통된 코드로 사용
5. 추상화
	- 상위 클래스/메소드/함수에서 간략하게 애플리케이션 특성 표현
	- 상세 내용은 하위 클래스/메소드/함수에서 구현
# 연결 리스트 (Linked List)
- 노드들이 순차적으로 연결되어 있어, 임의의 노드에 빠르게 접근하기 어려움
- 배열처럼 인덱스를 이용하여 바로 접근할 수 없기 때문에, 원하는 노드를 찾기 위해서는 연결된 노드를 따라 순차적으로 탐색 필요
- 검색 속도가 배열에 비해 느리다는 단점 보유
- 노드의 삽입이나 삭제가 쉬움
- 연결을 해주는 포인터(Pointer)를 위한 추가 공간 필요
- 연결 리스트 중에서 중간 노드 연결이 끊어지면 그 다음 노드 찾기 어려움

## 이진 검색 알고리즘
- 탐색 효율이 좋고 탐색 시간이 적게 소요된다.
- 검색할 데이터가 정렬되어 있어야 한다.
- 비교 횟수를 거듭할 때마다 검색 대상이 되는 데이터의 수가 절반으로 줄어든다.
### 이진 검색 방법
1. 대상 범위의 첫 번째 원소 위치를 Low로, 마지막 원소의 위치를 High로 두고서 중간 원소의 위치인 Mid를 (Low + High)/2로 구하기
2. 찾고자 하는 Key와 중간값을 비교
	- Key > 중간값 : Low를 Mid+1로 두고서 계속 수행
	- Key < 중간값 : High를 Mid-1로 두고서 계속 수행
	- Key = 중간값 : 검색 완료

### 선택 정렬 (Selection Sort)
1. 최소값을 찾아 첫 번째 숫자와 위치 교환
2. 이후 정렬된 값을 제외한 최소 숫자를 정렬되지 않은 숫자 중 첫 번째 숫자와 다시 위치 교환
3. 1~2 과정을 반복

### 삽입 정렬 (Insertion Sort)
1. 두 번째 숫자와 첫 번째 숫자를 비교하여 크기 순으로 정렬
2. 이후 3번째 숫자를 앞서 정렬된 숫자들 사이에 크기 순에 맞게 재정렬
3. 1~2 과정을 반복

- 오름차순 수행 시 매 회전마다 마지막 값이 가장 큰 값이 된다.
1. 왼쪽부터 인접한 두 숫자 간 크기 비교하여 위치 교환
2. 1 Pass마다 정렬되는 숫자들 중 가장 큰 숫자를 오른쪽으로 밀어서 배치

## 트리 순회 방법
![](https://velog.velcdn.com/images/kik328288/post/0fb171cd-f4e4-45e0-bda3-c3cd3904e6be/image.png)

- 순환 복잡도 (V(G)) = 화살표수(E) - 노드 수(N) + 2
### 트리의 전체 노드를 구하는 공식
- 전체 노드의 수 = 루트 노드 수 + 간 노드의 자식 노드 수 + 단말 노드 수 (자식이 없는 노드) + 트리의 차수 (한 노드가 가질 수 있는 최대 자식의 수)
# 해시 기법 (해시 함수)
- 임의의 길이의 입력 데이터를 받아 고정된 길이의 해시값으로 변환하는 기법
- 주어진 원문에서 고정된 길이의 의사 난수를 생성하며 생성된 값을 해시값이라 함
- 디지털 서명에 이용되며 데이터 무결성을 제공
- 블록체인에서 체인 형태로 사용되어 데이터의 신뢰성을 보장
- 해시 함수는 일방향 함수
- 종류 : SHA, SHA1, SHA256, RMD160, HAS-160, HAVAL 등
# 2과목 추가 개념
## 제품 관련
1. Vaporware
	- 제조사가 시장에 내놓겠다고 공표했지만 개발과 출시 일정이 계속 연기되면서, 향후에 출시되지 않을 가능성이 있는 제품
	- 출시가 지연되거나 취소될 가능성이 높은 제품
2. Hypeware
	- 과도한 홍보로 인해 소비자의 기대감을 부풀리는 제품
3. Wishware
	- 개발되기를 바라는 제품이지만 현실성이 낮은 제품
4. Unicorn
	- 혁신적이지만 실현 가능성이 낮은 제품
5. Blue Sky
	- 미래 지향적인 아이디어나 기술
## 스택을 이용한 연산
1. 재귀 호출
2. 후위 표기법 (Postfix)
3. 깊이 우선 탐색
4. 같이 왔던 길을 되돌아가는 경우
