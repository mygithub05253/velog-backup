# 데이터베이스
- 특정 조직 내 필요한 데이터들의 모임, 공용으로 소유/유지/이용하는 공용 데이터
## 데이터베이스 설계 단계

| 설계     | 설명                                                                                                                                                                                                    |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 개념적 설계 | 독립적인 개념 스키마 모델링<br>==트랜잭션 모델링==<br>산출물 : E-R 다이어그램                                                                                                                                                    |
| 논리적 설계 | 종속적인 논리 스키마 설계<br>==트랜잭션 인터페이스 설계==<br>정규화<br>논리적 데이터베이스 구조로 Mapping<br>스키마의 평가 및 정제(최적화)                                                                                                             |

## 데이터베이스 분할
1. 범위 분할 (Range Partition)
	- 연속적인 값을 기준으로 분할하는 방법
	- EX) 날짜를 기준으로 분할
2. 목록 분할 (List Partition)
	- 이산적인 값을 기준으로 분할하는 방법
	- EX) 국가를 기준으로 분할
3. 해시 분할 (Hash Partition)
	- 해시 함수를 사용해 분할하는 방법
	- EX) 고객 ID를 기준으로 분할
4. 합성 분할 (Composite Partition)
	- 두 가지 이상의 분할 방법을 결합하여 사용하는 방법
	- EX) 날짜를 기준으로 Range Partition, 국가를 기준으로 List Partiton 수행
## 데이터베이스 병행 제어 목적
1. 데이터베이스 공유 최대화
2. 데이터베이스 일관성 최대화
3. 시스템 활용도 최대화
4. 사용자에 대한 응답 시간 최소화
## 데이터베이스 무결성 규정 (Integrity Rule)
- 무결성 규정에는 데이터가 만족해야 될 제약 조건, 규정을 참조할 때 사용하는 식별자 등의 요소 포함 가능
- 무결성 규정의 대상 : 도메인, 키, 종속성 등
- 릴레이션 무결성 규정 : 릴레이션을 조작하는 과정에서의 의미적 관계를 명세한 것

## 릴레이션
1. 튜플과 속성은 유일하며 ==순서와 무관==
2. 튜플은 삽입/삭제 등에 의해 계속 변하며 서로 상이한 값을 가짐
3. 속성의 값은 분해 불가능하며 동일할 수 있음
4. 튜플을 식별하기 위해 속성(필드)의 일부를 Key로 설정
5. ==속성은 Null 값을 가질 수 있으나, 기본키에 해당되는 속성은 Null 값을 가질 수 없음==
### 릴레이션 특징
1. 튜플의 유일성 : 모든 튜플은 서로 다른 값을 가진다.
2. 튜플의 무순서성 : 하나의 릴레이션에서 튜플의 순서는 없다.
3. 속성의 원자성 : 속성값은 원자값을 갖고, 논리적으로 더 이상 분해될 수 없는 최소 단위 (오답 보기 : 물리적으로 분해할 수 없다고 나옴)
4. 속성의 무순서성 : 각 속성은 릴레이션 내에서 유일한 이름을 가지며, 속성의 순서는 큰 의미가 없음
# 분산 처리 시스템
- 다수의 사용자들이 데이터 공유 가능 및 사용자들 간 통신 용이
- 점진적인 확장이 가능하지만 보안 문제 발생 가능
- 개발 난이도가 높아 개발 비용 많이 소모
- 시스템 전체의 정책을 결정하는 통합적인 제어 기능 필요
- 집중형(Centralized) 시스템에 비해 소프트웨어 개발이 어려움
- 귀중한 장치들이 다수의 사용자들에 의해 공유 가능
- 종류 : 클라이언트/서버 모델, 프로세서 풀 모델, 혼합 모델
# 분산 데이터베이스
- 논리적으로는 하나의 시스템에 존재하지만 물리적으로는 연결된 다수의 컴퓨터에 분산되어 있는 데이터베이스
- ==사용자 입장에서 데이터베이스의 위치/중복/병행/분할/장애 여부 인식할 필요 없음==
## 분산 데이터베이스 구성 요소
1. 분산 처리기 (분산된 지역 컴퓨터)
2. 분산 데이터베이스
3. 통신 네트워크
## 분산 데이터베이스 목표
1. 위치 투명성 (Location)
2. 중복 투명성 (Replication)
3. 병행 투명성 (Concurrency)
4. 분할 투명성 (Division)
5. 장애 투명성 (Failure)
## 분산 데이터베이스 특징
1. 지역 서버의 고유 데이터에 대한 작업은 중앙 서버의 통제 없이 자유롭게 수행 가능
2. 새로운 지역 서버를 추가하거나 장비를 추가하는 등의 작업 용이
3. 논리적으로는 하나의 시스템에 속하지만 물리적으로는 여러 개의 컴퓨터 사이트에 분산되어 있는 형태
4. 위치 투명성, 중복 투명성, 병행 투명성, 장애 투명성을 목표로 함
5. 데이터베이스의 설계 및 소프트웨어 개발이 어려워서 개발 비용과 처리 비용 증가
# 키 (Key)

|           키            | 설명                                                                                                                                                                                                                                          |
| :--------------------: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 후보키<br>(Candidate Key) | 기본키로 사용 가능한 속성<br>모든 릴레이션에는 후보키 존재<br>==모든 튜플에 대해 유일성/최소성을 만족시켜야 함==<br>개체들을 고유하게 식별할 수 있는 속성                                                                                                                                               |
|  기본키<br>(Primary Key)  | 후보키 중에서 선택되어, ==중복된 값과 NULL 값 가질 수 없음==<br>유일성/최소성 만족하며 튜플 식별하기 위해 반드시 필요한 키                                                                                                                                                                |
| 대체키<br>(Alternate Key) | 후보키가 둘 이상일 때 기본키를 제외한 나머지 후보키                                                                                                                                                                                                               |
|   슈퍼키<br>(Super Key)   | 한 릴레이션 내 속성들의 집합으로 구성된 키<br>모든 튜플에 대해 ==유일성은 만족하지만, 최소성은 만족 불가능==<br>한 개 이상의 속성들의 집합으로 구성된 키                                                                                                                                                |
|  외래키<br>(Foreign Key)  | 다른 릴레이션의 기본키를 참조하는 속성 or 속성들의 집합<br>참조되는 릴레이션 기본키와 대응되어 릴레이션 간 참조 관계<br>현실 세계에 존재하는 개체 타입 간의 관계를 표현하는데 중요한 역할<br>외래키로 지정되면 참조 릴레이션의 기본키에 없는 값 입력 불가능<br>외래키를 포함하는 릴레이션 = 참조하는 릴레이션<br>대응되는 기본키를 포함하는 릴레이션 = 참조 릴레이션<br>참조 무결성 제약 조건과 밀접한 관계 |
- 유일성 : 하나의 키 값으로 하나의 튜플을 유일하게 식별
- 최소성 : 모든 레코드를 식별하는데 최소한의 속성으로만 구성
# 무결성 (Integrity)

|    무결성     | 설명                                                                     |
| :--------: | ---------------------------------------------------------------------- |
|   개체 무결성   | 기본키를 구성하는 어떤 속성도 NULL/중복 값 보유 불가능<br>기본키의 속성 값이 NULL 값이 아닌 원자 값을 갖는 성질 |
| 도메인/속성 무결성 | 릴레이션 내 튜플들이 각 속성의 도메인에 지정된 값만 보유                                       |
|   참조 무결성   | 외래키는 NULL 또는 참조 릴레이션의 기본키 값과 동일<br>릴레이션은 참조할 수 없는 외래키 값을 가질 수 없음       |
| 사용자 정의 무결성 | 속성 값들의 사용자가 정의한 제약 조건에 만족                                              |
| 데이터 무결성 강화 | 데이터 특성에 맞는 적절한 무결성을 정의하고 강화<br>강화 방법 : 제약 조건, 애플리케이션, 데이터베이스 트리거       |

# 정규화
- ==이상(Anomaly) 현상이 발생하지 않도록 중복성/종속성을 최소화==하기 위한 작업
- ==논리적 설계 단계==에서 수행
- 테이블을 분해하여 종속성을 줄이고 중복을 제거하는 것이 목적
- 속성 수가 적은 테이블로 분할되어 관리가 용이
- 데이터 구조 안정성 최대화
- 데이터 삽입 시 릴레이션 재구성 필요 최소화
## 이상 현상 종류 (Anomaly)
- 데이터의 중복으로 인해 릴레이션 조작 시 예상하지 못한 곤란한 현상이 발생하는 것

| 종류    | 설명                          |
| ----- | --------------------------- |
| 삽입 이상 | 불필요한 데이터가 함께 삽입             |
| 삭제 이상 | 필요한 데이터가 삭제                 |
| 갱신 이상 | 일부만 수정되어 데이터 불일치 → 정보 모순 발생 |
## 정규화 과정

|        과정        | 설명                                                                           |
| :--------------: | ---------------------------------------------------------------------------- |
|      제1정규형       | ==모든 도메인(Domain)이 원자 값==만으로 되어 있음                                            |
|      제2정규형       | 기본키가 아닌 속성이 기본키에 대한 ==완전 함수적 종속== 만족<br>==부분적 함수 종속을 제거==한 정규형               |
|      제3정규형       | 기본키 아닌 모든 속성이 기본키에 대해 ==이행 함수 종속 관계== 만족X<br>* 이행적 함수 종속 관계 : X→Y, Y→Z이면 X→Z |
| BCNF<br>(보이스/코드) | 릴레이션 R에서 ==모든 결정자가 후보키==인 정규형                                                |
|      제4정규형       | 릴레이션 R에 다치 종속이 성립하는 경우 <br>R의 모든 속성이 A에 함수적 종속 관계를 만족하는 정규형                  |
|      제5정규형       | 릴레이션 R의 모든 조인 종속이 R의 후보키를 통해서만 성립                                            |
# 함수적 종속 (Functional Dependency)
- 어떤 릴레이션 R에서 X와 Y를 각각 R의 속성(Attribute) 집합의 부분 집합일 경우, 속성(Attribute) X의 값 각각에 대해 시간에 관계없이 항상 속성(Attribute) Y의 값이 오직 하나만 연관되어 있을 때 Y는 X에 대해 함수 종속이라고 함
- 표현 : X → Y

# 데이터 사전 (시스템 카탈로그)
- 모든 데이터 객체에 대한 정의나 명세에 관한 정보를 유지/관리하는 시스템 테이블
- 데이터베이스에 대한 데이터인 메타데이터(Metadata)를 저장
- SQL 문을 이용해 내용 검색 가능
- 시스템이 자동으로 생성/유지하며, ==일반 사용자가 직접 내용 갱신 불가능==
- ==데이터 디렉토리(Data Directory)== : 데이터 사전 접근에 필요한 위치 정보 포함
- 시스템이 자신을 필요로 하는 스키마 및 여러 가지 객체에 관한 정보를 포함하고 있는 시스템 데이터베이스
- 시스템 카탈로그는 DBMS가 스스로 생성하고 유지
# 인덱스 (Index)
- ==데이터 레코드의 빠른 접근/조회==를 위해 별도로 구성한 순서 데이터
- 데이터베이스에서 테이블 삭제 시 인덱스도 같이 삭제 (==사용자가 직접 변경 가능==)
- 인덱스 생성 (CREATE) / 인덱스 삭제 (DROP)
## 인덱스 특징
- 문헌의 색인, 사전과 같이 데이터를 쉽고 빠르게 찾을 수 있도록 만든 데이터 구조
- 테이블에 붙여진 색인으로 데이터 검색 시 처리 속도 향상에 도움
- 대부분의 데이터베이스에서 테이블을 삭제하면 인덱스도 같이 삭제
- 인덱스는 수정 불가능하며 생성은 CREATE문, 삭제는 DROP문 사용
## 인덱스 종류
1. 클러스터
2. 넌클러스터
3. 트리 및 함수 기반
4. 비트맵 및 비트맵 조인
5. 도메인
# 트랜잭션
- 데이터베이스 상태를 변환시키는 ==하나의 논리적 기능을 수행하기 위한 작업 단위==
- ==COMMIT== : 트랜잭션 정상 종류 후 변경된 내용을 DB에 반영하는 명령어
- ==ROLLBACK== : 트랜잭션 비정상 종료 후 모든 변경 작업을 취소하고 이전 상태로 원복
- 트랜잭션이 부분 완료 상태에 도달했다가 실패 상태로 가는 경우는 트랜잭션이 데이터베이스에 반영되기 직전에 장애가 발생한 경우
- 디스크 출력 도중의 하드웨어 장애
	- 트랜잭션의 데이터베이스 반영을 방해할 수 있는 장애
	- 트랜잭션이 부분 완료 상태에 도달했다가 실패 상태가 되는 경우
## 트랜잭션 성질

|          성질          | 설명                                                                                                                                   |
| :------------------: | ------------------------------------------------------------------------------------------------------------------------------------ |
|  원자성<br>(Atomicity)  | 트랜잭션은 무조건 아래의 2가지 경우의 수만 존재<br>1. 모든 트랜잭션 연산이 정상적으로 수행 (Commit)<br>2. 어떠한 연산도 수행되지 않음 (Rollback)<br>완전하게 수행 완료되지 않으면 전혀 수행되지 않아야 한다. |
| 일관성<br>(Consistency) | 시스템의 고정 요소는 트랜잭션 수행 전/후로 동일                                                                                                          |
|  독립성<br>(Isolation)  | 개별 트랜잭션은 다른 트랜잭션의 간섭 및 영향을 받지 않아야 함<br>병행 실행 경우 어느 하나의 트랜잭션 실행 중 다른 트랜잭션 연산 수행 불가능                                                   |
| 영속성<br>(Durability)  | 완료된 트랜잭션 결과는 데이터베이스에 영구적으로 기록되어야 함                                                                                                   |
## 트랜잭션의 주요 상태
1. 활동 (Active)
	- 트랜잭션이 실행 중인 상태
2. 부분 완료 (Partially Committed)
	- 모든 명령이 실행되고, 마지막 단계에서 Commit을 기다리는 상태
3. 완료 (Committed)
	- 트랜잭션이 성공적으로 종료되어 변경 사항이 데이터베이스에 확정된 상태
4. 실패 (Failed)
	- 트랜잭션 도중 오류가 발생해 더 이상 정상 수행이 불가능한 상태
5. 철회 (Aborted)
	- 오류가 발생한 트랜잭션이 Rollback을 수행하여 이전 상태로 복구된 상태
# 뷰 (View)
- 기본 테이블에 기반을 둔 이름을 가지는 가상 테이블 (기본 테이블과 동일 형태)
- 저장 장치 내 가상으로, ==논리적으로 존재== (실존하는 ==물리적 존재 X==)
- 기본 테이블 및 뷰 삭제 시 해당 테이블/뷰를 기초로 정의된 다른 뷰도 자동 삭제
## 뷰의 특징
- 뷰의 생성 시 CREATE문, 검색 시 SELECT문을 사용
- 뷰의 정의 변경 시 ALTER문 사용 불가능하며 DROP문 이용
- 뷰를 이용한 또 다른 뷰의 생성(정의) 가능
- 하나의 뷰 제거 시 해당 뷰를 기초로 정의된 다른 뷰도 함께 제거
- 뷰가 정의된 기본 테이블이 제거되면 뷰도 자동적으로 제거
- 뷰에 대한 조작에서 삽입, 갱신, 삭제 연산은 제약이 따름
## 뷰의 장단점

| 장점                 | 단점                 |
| ------------------ | ------------------ |
| ==논리적 데이터 독립성 제공== | ==독립적 인덱스 보유 불가능== |
| 데이터 자동 보안 제공       | ALTER 변경 불가능       |
| 데이터 관리 용이          | 삽입, 삭제, 갱신, 연산 제약  |
# 병행 제어 기법 종류

|         종류         | 설명                                                                                                                  |
| :----------------: | ------------------------------------------------------------------------------------------------------------------- |
|       로킹 기법        | 일관성과 무결성을 유지하기 위한 트랜잭션의 순차적 진행 보장<br>데이터베이스에서 하나의 트랜잭션에 사용하는 데이터를 다른 트랜잭션이 접근하지 못하다록 잠그는 행위                         |
|       낙관적 검증       | 일단 트랜잭션을 수행하고, 트랜잭션 종료 시 검증을 수행<br>트랜잭션 수행 동안은 어떠한 작업도 수행하지 않음                                                      |
|     타임 스탬프 기법      | 타임 스탬프를 부여해 부여된 시간에 따라 트랜잭션 수행<br>데이터베이스에 접근하는 시간을 미리 정함<br>정해진 시간의 순서대로 데이터베이스에 접근하게 함<br>순차적이기 때문에 교착 상태가 발생하지 않음 |
| 다중버전 동시성 제어 (MVCC) | 타임스탬프를 비교해 직렬 가능성이 보장되는 적절한 버전을 선택해 접근하도록 함<br>충돌이 발생할 경우 복귀(Rollback) 수행                                           |
|     2단계 로킹 규약      | 1. 확장 단계 : 새로운 Lock은 가능하고 Unlock은 불가능<br>2. 축소 단계 : Unlock 가능, 새로운 Lock 불가능<br>   (순차적이지 않기 때문에 교착 상태 발생)           |
## 교착 상태 발생 조건

|             조건              | 설명                                                                                              |
| :-------------------------: | ----------------------------------------------------------------------------------------------- |
| 상호 배제<br>(Mutual Exclusion) | 한 번에 한 개의 프로세스만이 공유 자원 사용 가능<br>EX) 데커 알고리즘, Lamport의 빵집 알고리즘<br>Test and Set 기법, Swap 명령어 기법 등 |
|  점유와 대기<br>(Hold and Wait)  | 이미 자원을 가진 프로세스가 다른 자원의 할당을 요구한다.                                                                |
|   비선점<br>(Non-preemption)   | 프로세스에 할당된 자원은 사용이 끝날 때까지 강제로 빼앗을 수 없다.                                                          |
|  환형 대기<br>(Circular Wait)   | 이미 자원을 가진 프로세스가 앞이나 뒤의 프로세스의 자원을 요구                                                             |

## 로킹 단위 (Locking)
- 로킹 단위가 커지면 로크의 수가 적어 관리가 쉬워지지만, 병행성 수준이 낮아짐
- 로킹 단위가 작으면 로크의 수가 많아 관리가 어려워지지만, 병행성 수준이 높아짐
- 로킹의 대상이 되는 객체의 크기
- 로킹 : 데이터베이스 병행 제어 위해 트랜잭션(transaction)이 접근하고자 하는 데이터를 잠가(lock) 다른 트랜잭션이 접근하지 못하도록 하는 병행 제어 기법
- 대상 : 데이터베이스 / 파일 / 레코드

# SQL (Structured Query Language) 분류
## 데이터 정의어 (DDL : Data Definition Language)
- ==논리/물리적 데이터 구조 정의==

| 종류          | 설명                                                     |
| ----------- | ------------------------------------------------------ |
| CREATE (생성) | CREATE ==DOMAIN / SCHEMA / TABLE / VIEW / INDEX → 생성== |
| ALTER (변경)  | TABLE 이름 변경 → ALTER TABLE / ==컬럼 추가==                  |
| DROP (삭제)   | DROP ==DOMAIN / SCHEMA / TABLE / VIEW / INDEX → 삭제==   |
- 삭제 옵션
	- CASCADE : 참조하는 모든 개체 함께 제거
	- RESTRICTED : 제거할 요소를 다른 개체가 참조 시 제거 취소
## 데이터 조작어 (DML : Data Manipulation Language)

| 종류          | 설명                                    |
| ----------- | ------------------------------------- |

### DELETE문
- 테이블의 행을 삭제할 때 사용
- DELETE문은 테이블 내의 튜플들만 삭제하지만, DROP문은 테이블 자체를 삭제
- SQL을 사용 용도에 따라 분류할 경우 DML에 해당

## 데이터 제어어 (DCL : Data Control Language)
- ==데이터 보안/복구/병행 수행 제어 무결성 유지==

| 종류       | 설명                                                                                                                       |
| -------- | ------------------------------------------------------------------------------------------------------------------------ |
| COMMIT   | ==수행된 결과를 실제 물리적 디스크로 저장 후 작업의 정상 완료 통보==                                                                                |
| ROLLBACK | COMMIT 수행되지 않은 작업은 취소하고, 이전 상태로 원복                                                                                       |
| GRANT    | 데이터베이스 사용자에게 사용 권한 부여<br>GRANT 권한 리스트 ON 개체 TO 사용자 [WITH GRANT OPTION]<br>* WITH GRANT OPTION : 부여 받은 권한을 다른 사용자에게 재부여   |
| REVOKE   | 데이터베이스 사용자의 사용 권한 취소<br>REVOKE [GRANT OPTION FOR] 권한 리스트 ON 개체 FROM 사용자<br>* GRANT OPTION FOR : 다른 사용자에게 권한 부여 가능한 권한 취소 |
# 데이터 회복 기법

|     기법     | 설명                                                                                                                |
| :--------: | ----------------------------------------------------------------------------------------------------------------- |
|  즉시 갱신 기법  | 트랜잭션 실행 상태에서 변경되는 내용을 바로 데이터베이스에 적용<br>변경되는 모든 내용을 로그(Log)에 기록하여 장애 시 해당 로그 토대로 복원<br>Redo / Undo 모두 수행           |
|  지연 갱신 기법  | 트랜잭션 수행 후 부분 완료될 때까지는 데이터베이스에 바로 적용하지 않고, 지연 시킨 후 부분 완료 시 로그(Log)의 내용을 토대로 저장<br>트랜잭션이 실패할 경우 Undo 수행 없이 Redo만 수행 |
|  검사 시점 기법  | 트랜잭션 수행 중간에 검사 시점 (Checkpoint) 지정하여 검사 시점까지 부분 수행 후 완료된 내용을 중간중간 데이터베이스에 저장                                       |
| 그림자 페이징 기법 | 로그 미사용<br>데이터베이스를 동일 크기의 페이지로 분할 후 각 페이지마다 복사하여 그림자 페이지 보관<br>변경 내용은 원본 페이지에만 적용하여 장애 발생 시 해당 페이지 사용/회복           |
- Deferred Modification
	- 변경된 데이터를 실제로 디스크에 반영하는 것을 지연시키는 방식
	- 데이터의 논리적인 수정 작업을 기록하고, 나중에 특정 시점에 변경 사항을 일괄적으로 디스크에 반영하는 복구 기법과 관련
- Immediate Update
	- 데이터의 변경 사항을 즉시 디스크에 반영하는 방식
- Shadow Paging
	- 복구를 위해 일부 페이지를 원래 페이지와 별도의 그림자 페이지로 유지하는 방식
- Checkpoint
	- 특정 시점에서의 상태를 기록하는 것
# 파일의 구조
1. 순차 파일 (Sequential File)
	- 입력되는 데이터들을 논리적인 순서에 따라 물리적 연속 공간에 순서대로 기록하는 방식
2. 색인 순차 파일 (Indexed Sequential File)
	- 레코드들을 키 값 순으로 정렬 시켜 기록하고 레코드의 키 항목만을 모은 색인(인덱스)을 구성하여 편성하는 방식
3. 직접 파일 (Direct File)
	- 파일을 구성하는 레코드들을 특정 순서 없이 임의의 물리적 저장 공간에 기록하는 방식
	- 해싱 등의 사상 함수를 사용해 레코드 키에 의한 주소 계산을 통해 레코드를 접근할 수 있도록 구성한 파일
4. 역 파일 (Inverted File)
	- 특정 항목을 여러 개의 색인으로 만들어 항목별 특성에 맞게 작업할 수 있도록 하는 방식
5. 다중 리스트 파일 (Multi-List File)
	- 각 키에 대해 색인을 만든 다음 각 데이터 레코드들 간에 다중 리스트를 구축하여 구성하는 방식
6. 다중 링 파일 (Multi-Ring File)
	- 같은 특성을 가진 레코드들을 일련의 포인터로 연결하여 구성하는 방식
# 3과목 기본 용어
1. Concurrency Control (병행 제어, 동시성 제어)
	- 여러 사용자 또는 프로세스가 동시에 데이터베이스에 접근할 때 일관성을 유지하고 충돌을 방지하기 위한 제어 메커니즘
2. Normalization (정규화)
	- 데이터베이스 설계 과정에서 중복을 최소화하고 데이터의 일관성을 유지하기 위해 테이블을 구조화하는 작업
3. Transaction (트랜잭션)
	- 데이터베이스에서 하나의 논리적인 작업 단위
	- 일련의 데이터 조작 작업을 원자적(Atomic), 일관적(Consistent), 고립적(Isolated), 영구적(Durable)으로 수행하는 것을 보장
4. Trigger (트리거)
	- 데이터베이스에서 특정 이벤트가 발생할 때 자동으로 실행되도록 정의된 저장 프로시저
	- 데이터베이스 테이블에서 INSERT, UPDATE, DELETE 등의 이벤트가 발생할 때, 사전에 정의된 절차적 동작이 자동으로 실행되도록 구성된 객체
	- 특정 사건이 발생하면 연쇄적으로 다른 작업이 수행되도록 하는 기능
	- 이벤트가 발생할 때마다 관련 작업이 자동으로 수행되는 절차형 SQL
5. Relational Algebra (관계 대수)
	- 릴레이션(테이블)을 조작하기 위한 수학적 연산 집합
	- 원하는 결과를 얻기 위해 무엇을, 어떤 순서로 수행할 지 명시하는 절차적 언어
	- 일반 집합 연산과 순수 관계 연산으로 구분
	- 질의에 대한 해를 구하기 위해 수행해야 할 연산의 순서 명시
6. 이벤트 (Event)
	- 시스템에 어떤 일이 발생한 것
7. 프로시저 (Procedure)
	- 특정 기능을 수행하는 일종의 트랜잭션 언어
	- 호출을 통해 실행되어 미리 저장해 놓은 SQL 작업을 수행
8. 옵티마이저 (Optimizer)
	- DBMS에 내장되어 작성된 SQL이 효율적으로 수행되도록 최적의 경로를 찾아 주는 모듈
