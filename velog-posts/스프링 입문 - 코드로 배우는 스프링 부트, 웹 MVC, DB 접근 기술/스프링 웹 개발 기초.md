# 스프링 웹 개발 기초

---

스프링 부트에서 웹을 개발하는 방식은 크게 **정적 컨텐츠**, **MVC와 템플릿 엔진**, **API** 세 가지로 구분된다.

## 1. 정적 컨텐츠 (Static Content)

---

서버에서 별도의 가공이나 프로그래밍 로직 없이 파일을 있는 그대로 웹 브라우저에 전달하는 방식이다.

### 1.1 동작 원리

---

웹 브라우저가 요청을 보냈을 때 스프링 부트는 다음과 같은 순서로 리소스를 탐색한다.

1. **컨트롤러 우선순위:** 스프링 컨테이너에서 해당 요청(URL)을 처리할 컨트롤러(`@Controller`)가 있는지 먼저 찾는다.
2. **리소스 반환:** 컨트롤러가 매핑되어 있지 않다면, `resources/static` 경로 하위에서 해당 파일을 찾아 반환한다.

### 1.2 예제 코드

---

- **파일 경로:** `resources/static/hello-static.html`
- **HTML 작성:**

```html
<!DOCTYPE HTML>
<html>
<head>
    <title>static content</title>
    <meta http-equiv="Content-Type" content="text/html; charset=UTF-8" />
</head>
<body>
    정적 컨텐츠 입니다.
</body>
</html>
```

- **실행:** `http://localhost:8080/hello-static.html` 접속 시 HTML 파일 내용이 그대로 렌더링 된다.

## 2. MVC와 템플릿 엔진 (MVC & Template Engine)

---

서버 사이드에서 HTML을 동적으로 생성(렌더링)하여 클라이언트에게 전달하는 방식이다. 과거에는 View에 로직을 포함시켰으나, 현재는 역할에 따라 Model, View, Controller로 분리하여 개발한다.

### 2.1 구성 요소

---

- **Controller:** 비즈니스 로직을 처리하고, 화면에 필요한 데이터를 Model에 담아 View로 넘겨준다.
- **Model:** 화면(View)에서 사용할 데이터를 담는 객체이다.
- **View:** 화면을 그리는 역할(UI)에 집중한다.

### 2.2 동작 원리

---

1. **요청 수신:** 웹 브라우저가 요청(`localhost:8080/hello-mvc`)을 보낸다.
2. **Controller 실행:** 내장 톰켓 서버를 거쳐 스프링 컨테이너 내의 매핑된 Controller 메서드가 호출된다.
3. **반환 (Return):** Controller는 리턴 값으로 논리적인 뷰 이름(`hello-template`)을 반환하고, `Model`에 데이터(`name: spring`)를 담는다.
4. **ViewResolver 동작:** `viewResolver`가 리턴된 뷰 이름에 해당하는 템플릿 파일(`resources/templates/hello-template.html`)을 찾는다.
5. **렌더링:** 템플릿 엔진(Thymeleaf)이 템플릿 파일과 Model 데이터를 결합하여 HTML로 변환한 후 클라이언트에 응답한다.

### 2.3 예제 코드

---

**Controller (`HelloController.java`)**

```java
@Controller
public class HelloController {
    @GetMapping("hello-mvc")
    public String helloMvc(@RequestParam("name") String name, Model model) {
        model.addAttribute("name", name); // 파라미터로 받은 name을 모델에 담음
        return "hello-template"; // 템플릿 파일 이름 리턴
    }
}
```

**View (`resources/templates/hello-template.html`)**

```html
<html xmlns:th="http://www.thymeleaf.org">
<body>
    <p th:text="'hello' + ${name}">hello! empty</p>
</body>
</html>
```

## 3. API (Application Programming Interface)

---

HTML 같은 뷰 템플릿을 거치지 않고, HTTP 응답 본문(Body)에 데이터를 직접 실어서 보내는 방식이다. 주로 JSON 데이터 포맷을 사용한다.

### 3.1 핵심 애노테이션

---

- **`@ResponseBody`:** 뷰 리졸버(`viewResolver`)를 사용하지 않고, 반환된 데이터를 HTTP Body에 직접 입력하겠다는 의미이다.

### 3.2 동작 원리

---

1. **요청 수신:** 웹 브라우저가 요청(`localhost:8080/hello-api`)을 보낸다.
2. **Controller 실행:** `@ResponseBody`가 적용된 경우, 스프링은 `viewResolver` 대신 **`HttpMessageConverter`**를 동작시킨다.
3. **데이터 변환 및 응답 (HttpMessageConverter):**
    - **문자열 반환 시:** `StringHttpMessageConverter`가 동작하여 문자열 그대로를 반환한다.
    - **객체 반환 시:** `MappingJackson2HttpMessageConverter`가 기본으로 동작하여 객체를 **JSON** 포맷으로 변환해 반환한다.
    - 참고: 클라이언트의 HTTP Accept 헤더와 서버의 반환 타입 정보를 조합해 적절한 컨버터가 선택됨

### 3.3 예제 코드 (객체 반환)

**Controller (`HelloController.java`)**

```java
@Controller
public class HelloController {
    @GetMapping("hello-api")
    @ResponseBody // HTTP Body에 데이터 직접 반환
    public Hello helloApi(@RequestParam("name") String name) {
        Hello hello = new Hello();
        hello.setName(name);
        return hello; // 객체 반환 -> JSON으로 변환됨
    }

    static class Hello {
        private String name;

        public String getName() {
            return name;
        }
        public void setName(String name) {
            this.name = name;
        }
    }
}
```

**실행 결과 (`http://localhost:8080/hello-api?name=spring`)**

```json
{
    "name": "spring"
}
```

## 4. 비즈니스 요구사항 및 설계 (예시 : 회원 관리)

---

### 4.1 일반적인 웹 애플리케이션 계층 구조

---

스프링 웹 애플리케이션은 보통 다음과 같은 정형화된 패턴으로 구성된다.

![](https://velog.velcdn.com/images/kik328288/post/0854f21c-aacf-457c-bc9b-0c5706f44ab3/image.png)

- **컨트롤러 (Controller):** 웹 MVC의 컨트롤러 역할을 하며, 외부 요청을 받아 서비스나 리포지토리를 호출한다.
- **서비스 (Service):** 핵심 비즈니스 로직(예: 중복 가입 방지, 회원 등급 관리 등)을 구현한다.
- **리포지토리 (Repository):** 데이터베이스에 접근하여 도메인 객체를 저장하고 관리하는 역할을 한다.
- **도메인 (Domain):** 회원, 주문, 쿠폰 등 데이터베이스에 저장되고 관리되는 비즈니스 객체이다.

### 4.2 요구사항 분석

---

- **데이터:** 회원 ID(시스템 식별자), 이름.
- **기능:** 회원 등록, 회원 조회.
- **가정:** 아직 데이터 저장소(DB)가 선정되지 않은 가상의 시나리오이다(RDB, NoSQL 등 고민 중).

### 4.3 설계 전략

---

- 데이터 저장소가 확정되지 않았으므로, 나중에 구현체를 쉽게 교체할 수 있도록 **인터페이스(Interface)**를 정의하여 개발한다.
- 초기 개발 단계에서는 가벼운 **메모리 기반의 저장소(`MemoryMemberRepository`)**를 구현체로 사용한다.
